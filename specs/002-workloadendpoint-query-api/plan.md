# Implementation Plan: WorkloadEndpoint Query API with MAC Address

**Branch**: `002-we-api-mac-vlan` | **Date**: 2026-04-27 | **Spec**: [spec.md](./spec.md)

**Note**: This template is filled in by the `/speckit.plan` command. See `.specify/templates/plan-template.md` for the execution workflow.

## Summary

Extend Spiderpool Agent's WorkloadEndpoint query API to support retrieving complete network allocation details (IP/VLAN/MAC) via Unix Socket. The primary requirement is to enable external systems - particularly CNI plugins in public cloud integration scenarios - to query Pod IP allocation details including MAC addresses after IPAM delegates allocation to external providers.

Technical approach: Extend SpiderEndpoint CRD with optional `mac` field, accept MAC in IPAM request parameters, complete the `/v1/workloadendpoint` GET API with proper request/response schema, and expose via Unix Socket for external access.

## Phase 0: Research & Outline

### Unknowns to Resolve

1. **MAC Address Source**: Where does MAC come from?
   - **Research**: In public cloud integration, external provider returns MAC with IP
   - **Decision**: MAC provided in `IpamAddArgs` request parameter
   
2. **API Response Schema**: What fields to include?
   - **Research**: Existing SpiderEndpoint structure has IPs, VLAN, pools, gateways, routes
   - **Decision**: Add MAC, keep same structure, omit when not set
   
3. **Unix Socket Path**: Where should socket be located?
   - **Research**: Existing spiderpool-agent uses `/var/run/spidernet/spiderpool.sock`
   - **Decision**: Reuse existing socket, no new path needed

### Research Summary

| Topic | Decision | Rationale |
|-------|----------|-----------|
| MAC storage | SpiderEndpoint CRD | Needs persistence, already have CRD |
| MAC input | IpamAddArgs parameter | CNI/cloud provider passes MAC during allocation |
| API endpoint | Extend existing `/workloadendpoint` | Already has stub, needs completion |
| Field visibility | Omit when unset | Clean API, backward compatible |
| Socket access | Existing Unix socket | Reuse infrastructure |

## Phase 1: Design & Contracts

### Data Model

**IPAllocationDetail** (extends existing):
```yaml
interface: string        # NIC name
ipv4: string            # IPv4 with CIDR
ipv6: string            # IPv6 with CIDR
mac: string (optional)  # MAC address, e.g., "aa:bb:cc:dd:ee:ff"
vlan: int (optional)    # VLAN ID
ipv4Pool: string        # Pool name
ipv6Pool: string
ipv4Gateway: string
ipv6Gateway: string
routes: []Route
cleanGateway: bool
```

**WorkloadEndpointStatus** (API response):
```yaml
podNamespace: string
podName: string
podUID: string
node: string
interfaces: []InterfaceDetail
```

### API Contract

**GET /v1/workloadendpoint**

Parameters:
- `podNamespace` (query, required): Pod namespace
- `podName` (query, required): Pod name

Responses:
- `200`: Success with WorkloadEndpointStatus
- `404`: SpiderEndpoint not found
- `400`: Invalid parameters
- `500`: Internal error

### Quick Start (for API consumers)

```bash
# Query via Unix Socket
curl --unix-socket /var/run/spidernet/spiderpool.sock \
  "http://localhost/v1/workloadendpoint?podNamespace=default&podName=my-pod"

# Response (with MAC/VLAN):
{
  "podNamespace": "default",
  "podName": "my-pod",
  "podUID": "abc-123",
  "node": "node-1",
  "interfaces": [{
    "interface": "eth0",
    "ipv4": "10.0.0.1/24",
    "mac": "aa:bb:cc:dd:ee:ff",
    "vlan": 100
  }]
}
```
